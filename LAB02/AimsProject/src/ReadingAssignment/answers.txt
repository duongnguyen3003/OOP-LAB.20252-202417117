1. Constructor Overloading
Question: If you create a constructor method to build a DVD by title then create a constructor method to build a DVD by category. Does JAVA allow you to do this?
Answer: No, Java does not allow this. Method and constructor overloading requires the parameter list (the signature) to be different in terms of type, number, or order. Because both title and category are of type String, constructors like public DigitalVideoDisc(String title) and public DigitalVideoDisc(String category) would have the exact same signature: (String). The Java compiler would not be able to tell them apart and will throw a compile-time error.

2. Method Overloading (Varargs vs. Array)
Question: Try to add a method addDigitalVideoDisc which allows to pass an arbitrary number of arguments for dvd. Compared to an array parameter. What do you prefer in this case?
Answer: Passing an arbitrary number of arguments (using Java varargs, e.g., DigitalVideoDisc... dvds) is generally preferred over an array parameter for convenience and readability. It allows the caller to pass a comma-separated list of items directly into the method (like addDigitalVideoDisc(dvd1, dvd2, dvd3)) without forcing the user to explicitly write the code to construct an array object first. Under the hood, Java actually handles varargs by packing them into an array anyway, so it gives you the same functionality with a much cleaner syntax.

3. Passing Parameters
Question: Is JAVA a Pass by Value or a Pass by Reference programming language?
Answer: Java is strictly a Pass by Value programming language.

Question: After the call of swap(jungleDVD, cinderellaDVD) why does the title of these two objects still remain?
Answer: Because Java passes objects by value, it actually passes copies of the references (memory addresses) to the objects, not the objects themselves. When the swap method executes, it only swaps its local, copied reference variables. The original jungleDVD and cinderellaDVD reference variables back in the main method remain completely untouched and continue pointing to their original objects.

Question: After the call of changeTitle(jungleDVD, cinderellaDVD.getTitle()) why is the title of the JungleDVD changed?
Answer: The changeTitle method receives a copy of the reference pointing to the jungleDVD object. While it is a copy, it still points to the exact same physical object in the computer's memory (the heap). When dvd.setTitle(title) is called, it uses that reference pointer to reach into the object and modify its internal data. (Note: The line dvd = new DigitalVideoDisc(oldTitle) inside the method only reassigns the local copied reference to a new object, which does nothing to the original object outside the method).
